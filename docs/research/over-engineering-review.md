## Executive summary

1. **Delete `.models` tiers and `model-registry`; keep one profile-shaped runtime config.** Profiles already match the Pi launch contract (`provider`, `model`, `thinking`, `tools`), while tiers/registry add a second and third path with incompatible provider semantics and drifted defaults (`plugins/cmux-todo-board/bin/board-config:5-10`, `plugins/cmux-todo-board/bin/board-config:101-146`, `plugins/cmux-todo-board/bin/board-model:596-720`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:38-57`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:67-90`).
2. **Delete dead prompt layer `prompts/pi/init.md`.** The design doc says Pi reads it as the first message, but the spawn path only appends `common-system.md` and `roles/$ROLE.md`; `init.md` is not wired into launch (`plugins/cmux-todo-board/prompts/pi/init.md:1-3`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:35-42`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:80-97`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:159-179`).
3. **Delete or formalize the four extra profiles.** Code accepts nine profiles, but the prompt asset layout and profile docs define only five role files, so `backend-fast`, `repo-scout`, `test`, and `tiny-patch` are config surface without matching prompt assets/docs (`plugins/cmux-todo-board/bin/board-config:9-10`, `plugins/cmux-todo-board/docs/pi-profiles.md:1-27`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:19-23`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:33-42`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:163-166`).
4. **Simplify the Pi-only worker runtime by removing legacy multi-backend shims.** `--agent` / `--kind`, `agent_kind_*`, and provider-to-agent-kind routing are still carried even though only `pi` is supported (`plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/lib.sh:57-79`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:5-15`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:21-35`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-send.sh:5-12`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-send.sh:16-27`).
5. **Merge duplicated default tables; today there are already three conflicting copies.** Tier defaults disagree between `board-config`, `board-model list`, and the delegation policy doc, which means the human-facing docs/output already lie about runtime behavior (`plugins/cmux-todo-board/bin/board-config:5-6`, `plugins/cmux-todo-board/bin/board-model:744-789`, `plugins/cmux-todo-board/docs/delegation-policy.md:7-15`).

## The three-config-systems analysis

### What each system actually does

#### 1. Tiers (`.models`, `board-config --get-model`)

- Tiers are defined as five fixed labels in `board-config` and a default model table: `flash`, `pro`, `review`, `simple`, `top` (`plugins/cmux-todo-board/bin/board-config:5-6`).
- `resolve_model_full()` reads `.models.<tier>`, optionally resolves the value through `model-registry`, otherwise treats it as a direct model ID, then falls back to the built-in default table (`plugins/cmux-todo-board/bin/board-config:101-146`).
- `agent-spawn.sh` has a dedicated tier branch: if the third positional argument is one of those five labels, it shells out to `board-config` up to three times (`--get-model`, `--provider`, `--effort`) before launch (`plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:67-90`).
- The policy docs also treat tiers as the public delegation vocabulary (`plugins/cmux-todo-board/docs/delegation-policy.md:7-23`, `plugins/cmux-todo-board/docs/ORCHESTRATOR.md:18`).

**Net:** tiers are the policy-facing config path, but they are not the best match for Pi launch because they do not natively carry `thinking`, `tools`, or prompt-role selection.

#### 2. Profiles (`.profiles`, `board-config --get-profile`)

- Profiles are defined as named Pi launch presets in `DEFAULT_PROFILES_JSON`, each with `provider`, `model`, `thinking`, and `tools` (`plugins/cmux-todo-board/bin/board-config:9-10`).
- `resolve_profile_full()` deep-merges `.profiles.<name>` overrides on top of those defaults and returns the full launch object (`plugins/cmux-todo-board/bin/board-config:160-205`).
- `agent-spawn.sh --profile <name>` uses that full object directly: `MODEL=provider/model`, plus separate `thinking` and `tools` fields (`plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:38-57`).
- The prompt system also hangs off this path because spawn loads `roles/$ROLE.md`, with `ROLE` set from `PROFILE` (`plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:163-166`).

**Net:** profiles already match the actual Pi worker contract.

#### 3. Registry (`.model-registry`, `board-model add/edit/delete/asign`)

- Registry entries are CRUD-managed by `board-model add`, `edit`, and `delete` (`plugins/cmux-todo-board/bin/board-model:227-408`).
- `asign` maps a registry name into a tier by writing `.models.<tier> = <registry-key>` (`plugins/cmux-todo-board/bin/board-model:410-445`).
- `select --role` bypasses the registry and writes straight to `.profiles.<role>`; `select --tier` bypasses the registry and writes straight to `.models.<tier>` (`plugins/cmux-todo-board/bin/board-model:696-719`).
- So the registry is not a universal worker catalog; it is only an optional indirection layer under the tier system.

**Net:** registry exists to make tiers richer, not to serve profiles, and therefore doubles complexity without becoming the single source of truth.

### Where they conflict

1. **Tiers and profiles both choose a model/provider, but only profiles choose the rest of the Pi launch contract.** That means the system has two runtime entry points for the same conceptual decision (`plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:38-57`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:67-90`).
2. **Registry provider semantics are stale for Pi.** Registry validation only allows `opencode|codex`, while profile defaults and role selection use real Pi providers like `opencode-go`, `openai-codex`, and `anthropic` (`plugins/cmux-todo-board/bin/board-model:7-8`, `plugins/cmux-todo-board/bin/board-model:124-132`, `plugins/cmux-todo-board/bin/board-config:9-10`).
3. **The human-facing tables already disagree.** `board-config` defaults, `board-model list`, `docs/delegation-policy.md`, and `docs/pi-profiles.md` are not describing one shared runtime anymore (`plugins/cmux-todo-board/bin/board-config:5-10`, `plugins/cmux-todo-board/bin/board-model:744-789`, `plugins/cmux-todo-board/docs/delegation-policy.md:7-15`, `plugins/cmux-todo-board/docs/pi-profiles.md:6-12`).
4. **Profiles are also internally split.** `board-config` defines nine profile names, but the prompt-layering doc and prompt asset layout only define five role files (`plugins/cmux-todo-board/bin/board-config:9-10`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:19-23`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:33-42`).

### Recommendation

**Do not add a fourth persistent abstraction (`.workers`) before deleting the existing extras.** The honest simplification is:

1. **Keep one profile-shaped runtime config** because it already matches what Pi launch needs: `provider`, `model`, `thinking`, `tools`, plus a role name for prompt selection (`plugins/cmux-todo-board/bin/board-config:160-205`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:38-57`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:163-166`).
2. **Delete `.models` tiers and `.model-registry` as stored config.** They are the source of the provider-schema mismatch and the default-table drift (`plugins/cmux-todo-board/bin/board-config:101-146`, `plugins/cmux-todo-board/bin/board-model:227-445`, `plugins/cmux-todo-board/bin/board-model:596-719`).
3. **Keep tier names only as policy vocabulary in docs/UI, not as a second persistence model.** If the orchestrator wants `flash` or `pro` as routing hints, map those labels to supported profiles in one place instead of giving them their own config tree (`plugins/cmux-todo-board/docs/delegation-policy.md:7-23`, `plugins/cmux-todo-board/docs/ORCHESTRATOR.md:18`).
4. **Prune the profile list down to the documented/prompted set before any rename.** If a later rename from `.profiles` to `.workers` is still desired, do it as a rename-in-place after deleting tiers/registry, not as a fourth live abstraction (`plugins/cmux-todo-board/bin/board-config:9-10`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:33-42`).

**Why this over issue #138's "unify by adding" direction:** today the main problem is not lack of an abstraction; it is too many partially overlapping ones. Adding `.workers` first would preserve the tier path, preserve the registry path, preserve the profile path, and add migration glue on top. The plugin needs less config surface, not another namespace.

## What is NOT over-engineered

1. **`poll-wait.sh`'s event+poll fallback is justified.** The standby rule explicitly forbids active pane polling and wants event-driven waiting, but `poll-wait.sh` still keeps `poll-push.sh` as a fallback when the event path is unavailable. That redundancy is operational resilience, not abstraction for its own sake (`plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/poll-wait.sh:1-25`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/poll-wait.sh:72-169`, `plugins/cmux-todo-board/docs/ORCHESTRATOR.md:106-113`).
2. **Prompt layering with `common-system.md` + per-role files is justified.** The assets are small, role-focused, and loaded directly into the worker rather than copied into every task spec or orchestrator turn. The problem is the dead `init.md`, not the two-layer prompt structure itself (`plugins/cmux-todo-board/docs/pi-prompt-layering.md:11-29`, `plugins/cmux-todo-board/docs/pi-prompt-layering.md:45-55`, `plugins/cmux-todo-board/skills/cmux-agent-workflows/scripts/agent-spawn.sh:159-167`).
3. **Lite/full skill pairs are justified where they actually save context.** The diagnostics doc explicitly calls out oversized skills as a recurring token cost, and the `*-lite` variants are a direct response to that. I would trim thin wrapper skills before cutting the lite/full pattern (`plugins/cmux-todo-board/docs/orchestrator-diagnostics.md:36-42`, `plugins/cmux-todo-board/skills/board-onboard-lite/SKILL.md:8-17`, `plugins/cmux-todo-board/skills/cmux-agent-workflows-lite/SKILL.md:8-24`).
